# Puenteo — предложения по улучшению

**Автор:** Claude (сессия harbor-framework), 2026-07-22
**Версия puenteo на момент отзыва:** 0.4.2
**Контекст:** реальное использование — нужно было найти чужую claude-сессию, работавшую в
`~/dev/harbor-datasets`, и вытащить из неё, что там построено (датасет `polymorph-nav`).

---

## TL;DR

Инструмент **сделал работу за 4 команды** и без чтения гигабайтных `.jsonl` вручную —
это главная ценность. Общая оценка удобства для агент-к-агенту хэндовера: **9/10**.
Ниже — что зашло, что мешало, и конкретные предложения (сгруппированы по приоритету).

---

## Что реально сработало (сохранить как есть)

1. **`status` как первый экран.** Мгновенно показал все провайдеры (claude 52, codex 14,
   grok 6, pi 85) + пути + форматы экспорта + примеры команд. Идеальная точка входа —
   не пришлось гадать, где что лежит.
2. **`pull ... --query`** — киллер-фича. Отдаёт **компактный контекст-пак** (релевантные
   сообщения по запросу), а не весь транскрипт. Именно то, что нужно другому агенту:
   контекст без переполнения. Заголовок пака с `cwd`/`session_id`/`source` +
   дисклеймер «do not assume files still match disk» — очень грамотно.
3. **`list --json`** — машиночитаемо, легко фильтровать по `cwd`/summary в один pipe.
   Нашёл единственную нужную сессию из 30 мгновенно.
4. **Мульти-провайдер из коробки** (claude/codex/grok/pi) — не нужно знать формат
   каждого агента.
5. **`asb`/`pto`/`puenteo` алиасы** — приятная мелочь, ergonomично.

## Проблемы и предложения

### P0 — задело прямо в работе

**1. `search` без `--session` ищет по ВСЕМ сессиям и подмешивает мусор.**
Когда я искал `"polymorph-nav MCP ..."`, топ-результатов оказались из **моей текущей**
сессии (та же тема), а не из целевой. Пришлось переспрашивать с `--session`.
- **Предложение:** если задан контекст одной сессии (или пользователь только что сделал
  `pull/show <id>`), делать `search` по умолчанию **в рамках этой сессии**, а глобальный
  поиск — по явному флагу `--all`. Либо: в выдаче `search` **всегда** первой колонкой
  показывать `session_id` крупно и **группировать по сессии** (сейчас `session_id` есть,
  но результаты вперемешку и легко перепутать источник).
- **Ещё:** флаг `--exclude-session <id>` (исключить текущую свою сессию из глобального
  поиска — частый кейс, когда агент ищет «у соседа, не у себя»).

**2. Не хватает фильтра по `cwd`/проекту на уровне CLI.**
Я нашёл сессию через `list --json | python -c "...фильтр по cwd..."`. Это работает, но
типовой запрос «сессии, которые трогали каталог X» заслуживает первого класса.
- **Предложение:** `puenteo list --cwd ~/dev/harbor-datasets` (подстрока/glob по рабочему
  каталогу) и/или `--path-touched <glob>` (сессии, где агент читал/писал файл по маске).
  Это ровно то, как агенты ищут «кто работал над этим проектом».

### P1 — заметно улучшит хэндовер

**3. `title` часто бесполезен** (`<local-command-stdout>Set model to Opus 4.8 ...` с
ANSI-кодами `[1m`). Он берётся из первого стдаута слэш-команды, а не из сути диалога.
- **Предложение:** генерировать `title`/`summary` из **первого пользовательского запроса**
  или дешёвой выжимки; и **стрипать ANSI** в любых текстовых полях (сейчас `[1m`/`[22m`
  протекают в вывод — мешает и глазам, и парсингу).

**4. `pull --query` иногда тянет нерелевантный «хвост» (последние сообщения) вместо
самых релевантных.** В одном pull первым шёл `#190` (media_player), хотя запрос был про
итоговое состояние — релевантные `#311/#312` (развилка/решения) были ниже.
- **Предложение:** ранжировать строго по релевантности к query (а не по позиции), и дать
  `--top-k N` + `--max-chars` для явного контроля бюджета пака. Плюс режим
  `--mode decisions` — вытащить именно «решения/итоги/выводы» (сообщения, где агент
  фиксирует результат), это самый частый вопрос при хэндовере.

**5. Нет «карты» сессии.** Прежде чем `pull`, хочется быстрый скелет: сколько сообщений,
временной диапазон, ключевые вехи, менявшийся `cwd`.
- **Предложение:** `puenteo outline <id>` — таймлайн-оглавление (N сообщений, вехи по
  эвристике «assistant зафиксировал результат/ошибку/решение»), чтобы прицельно потом
  `pull --around <msg#>`.

**6. `pull --around <msg#>` / `show --range A:B`.** Найдя интересное сообщение через
`search` (напр. `#500`), хочется контекст ±N вокруг него, а не весь транскрипт и не
только `--tail`.

### P2 — приятно иметь

7. **`--since`/`--until` по времени** и сортировка `list` по времени последней активности
   (сейчас newest-first — хорошо, но фильтра по датам нет).
8. **Дедуп/связывание сессий одного проекта** — если один агент продолжал работу в
   нескольких сессиях по одному `cwd`, показать их как цепочку (`--group-by cwd`).
9. **Python API-примеры в `status`/README.** `status` намекает на «library + CLI», но
   агенту полезен готовый сниппет: `from puenteo import ... ; sessions = list_sessions(cwd=...)`.
10. **`export --query`** — экспортировать не весь транскрипт, а только релевантный срез
    в md/html (сейчас `export` — полный, `pull` — компактный текст; хочется компактный
    экспорт в файл для отчёта).
11. **Стабильные короткие ID.** Полные UUID длинные; `list` мог бы принимать и
    разрешать однозначный префикс (`pull 3627012b`) — <b>это уже работает</b>, стоит
    задокументировать явно (я угадал, но не был уверен).

## Идеальный сценарий хэндовера (как я бы хотел, чтобы это выглядело)

```bash
puenteo status                                  # что вообще есть
puenteo list --cwd ~/dev/harbor-datasets        # сессии по проекту  (P0 #2)
puenteo outline <id>                             # скелет сессии       (P1 #5)
puenteo pull <id> --mode decisions --top-k 15    # итоги/решения       (P1 #4)
puenteo search 'grep-negative' --session <id>    # прицельно внутри    (P0 #1)
puenteo pull <id> --around 500                   # контекст вокруг     (P1 #6)
```

Сейчас достижимо всё, кроме `--cwd`, `outline`, `--mode decisions`, `--around` — их
пришлось эмулировать через `list --json | grep` и повторные `pull`. Добавление этих
четырёх закрыло бы 90% сценариев «агент забирает контекст у другого агента».

## Вывод

puenteo уже сейчас — **правильный инструмент для agent-to-agent контекст-хэндовера**:
compact `pull`, мульти-провайдер, json-выдача. Доработки выше — про **прицельность**
(искать в одной сессии, фильтровать по проекту, вытаскивать именно решения) и
**гигиену вывода** (ANSI, осмысленные title). С ними это будет 10/10.
